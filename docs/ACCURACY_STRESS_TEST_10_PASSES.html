Pass 1Temporal accuracy: the dimension we completely missedPhysics: time as a variable
MISSINGAI accuracy degrades over time. A rubric ingested 6 weeks ago may have been updated by the lecturer. AURA still references the old version. No staleness detection. No "this document was ingested 42 days ago, confirm it is still current."
MISSINGProfile accuracy decays. A learner's accessibility needs, energy patterns, and cognitive load profile change over a semester. A profile set in Week 1 is increasingly inaccurate by Week 12. No time-decay model on profile fields. No "this profile is 90 days old, do any fields need updating?"
MISSINGPareto Steps become stale mid-task. A student starts an essay in Week 3, does not return until Week 7. The Pareto Steps were computed from a rubric that may have been superseded by an addendum. No version-stamping on Pareto Step generation.
MISSINGSession context drift. AURA uses last_session data. But if a student uses the system daily for a month, the last_session reference keeps moving and AURA loses the longer arc. No longitudinal accuracy model beyond the last session.
FIXEvery ingested document, every generated Pareto Step, and every profile field needs a created_at, last_verified_at, and expires_at. AURA checks staleness before every call and surfaces it if exceeded.
Pass 2Cascading error propagation: one wrong thing poisons everything downstreamEngineering: failure mode analysis (FMEA)
MISSINGIngestion error propagates silently to Pareto Steps. If the rubric is extracted with 40% confidence, the Pareto Steps built from it are wrong. But the Pareto Step confidence watermark only shows per-step, not "these steps were built from a low-confidence rubric." The root cause is invisible.
MISSINGWrong document type classification cascades. If a biology exam paper is classified as an "essay brief," every AURA response is structured around essay scaffolding, not exam technique. The classification error multiplies across the entire session.
MISSINGProfile field error cascades. If "processing_speed_multiplier" is set to 1.0 by default because the learner skipped that question, but the learner is actually a 3x processor, every idle threshold fires three times too fast. One missing field creates a session-long experience failure.
MISSINGNo error propagation graph. We log four isolated problems. We do not log the causal chain: "rubric confidence 0.4 → Pareto Steps unreliable → AURA guidance misdirected → learner worked on wrong section → submission underperformed."
FIXEvery log entry needs a parent_event_id field. If a Pareto Step was generated from a low-confidence ingestion, that link is stored. If an AURA response was built from a stale Pareto Step, that link is stored. The full error chain becomes queryable.
Pass 3Confirmation bias in the accuracy system itselfCognitive science + adversarial testing
MISSINGThe learner flag system only captures errors the learner notices. A learner who does not know the rubric was misread will not flag it. The accuracy system has a blind spot exactly where the learner has a blind spot. They reinforce each other.
MISSINGAURA confidently wrong is more dangerous than AURA uncertain wrong. If AURA says "your rubric requires critical analysis worth 30%" with certainty, and the actual rubric says 15%, the learner trusts the wrong number because AURA sounded sure. We measure whether the learner flags it. We do not measure calibration: does AURA's expressed certainty match its actual accuracy?
MISSINGThe four problems assume honest users. An adversarial user who deliberately uploads a fake rubric or a corrupted PDF breaks every downstream accuracy assumption. No adversarial input detection.
MISSINGNo ground truth comparison. We have no mechanism to compare AURA's extraction against a known-good version of the same document. Without ground truth we are measuring self-reported accuracy.
FIXBuild a calibration score per AURA response: does the expressed confidence (hedging language, confidence watermark) match the actual rubric_confidence and brieftext_length? Add automated weekly accuracy sampling: randomly select 50 ingestion events, re-run extraction, compare to stored result. Flag divergence above 15%.
Pass 4Accuracy at the edges: the 1% of users who break everythingChaos engineering + boundary condition testing
MISSINGZero-byte documents. Learner uploads an empty PDF (happens when a printer driver creates an empty file). pdfjs extracts nothing. confidence_score is 0. But the current system does not handle the "extracted nothing at all" case separately from "extracted with low confidence." AURA fires with empty briefText and hallucinates.
MISSINGMultilingual documents. A Chinese student uploads a Mandarin rubric. pdfjs extracts the characters but AURA's accuracy logging assumes English text hashing. The raw_text_hash is meaningless and the extracted_fields JSON is garbage. No multilingual extraction accuracy path.
MISSINGHandwritten or photo documents. A student photographs a handwritten feedback sheet. OCR runs. But OCR accuracy on handwriting is 60-80% at best. We have no separate logging path for OCR-sourced content vs clean PDF extraction. They both get the same confidence score calculation which is wrong for OCR.
MISSINGExtremely long documents. A 200-page thesis has its first 5000 chars hashed. But the relevant rubric criteria are on page 187. The accuracy logging does not reflect which pages were actually used for extraction.
FIXAdd extraction_method field: clean_pdf, ocr_image, ocr_handwritten, empty, multilingual. Each method gets its own confidence baseline and its own accuracy threshold. A 0.7 confidence for clean PDF extraction is very different from 0.7 for OCR handwriting.
Pass 5Distributed systems accuracy: what happens between servicesSystems engineering: CAP theorem + eventual consistency
MISSINGRace condition between IndexedDB write and Supabase sync. A learner saves a block at 11:58pm and closes the laptop. IndexedDB write succeeds. Supabase sync is in flight. Laptop goes offline. The data exists locally but the cloud has a 45-second-old version. The accuracy logging records "write confirmed" but the cloud version is wrong.
MISSINGThe four problems assume a single source of truth. In reality there are three: IndexedDB (local), Supabase (cloud), and the AURA API's in-memory context object. These three can diverge. We log writes to IndexedDB and we log the AURA context snapshot. We do not log whether they agreed with each other at the moment the API call fired.
MISSINGSupabase RLS (Row Level Security) silent failures. If an RLS policy is misconfigured, a write appears to succeed on the client but is silently rejected server-side. The learner sees "saved" but the data is not there. Our write verification hashes the local copy, not the server copy.
MISSINGAPI response truncation. The Anthropic API has token limits. If the AURA system prompt plus context plus conversation history exceeds the limit, the context is silently truncated. AURA responds without the full profile, without all rubric criteria, without the full brief. We have no logging for context window truncation events.
FIXAdd three fields to aura_context_snapshots: estimated_token_count, context_truncated (bool), truncated_fields (array). Before every API call, estimate the token count. If approaching the limit, log which fields were dropped and in what order. Surface to Aaron's dashboard as a P1 alert when truncation exceeds 5% of calls.
Pass 6Human accuracy: the learner is also a source of errorHuman factors engineering + UX research
MISSINGLearner confirms an incorrect extraction. The ingestion pipeline shows the learner what it extracted and asks them to confirm. But a learner who does not understand the rubric will confirm wrong extractions as correct. user_confirmed: true is not the same as extraction_accurate: true.
MISSINGLearner inputs incorrect profile data. A learner with undiagnosed ADHD selects "no attention difficulties" during onboarding. The profile is accurate to what they said, inaccurate to their actual needs. We have no mechanism to detect profile-behaviour divergence over time.
MISSINGLearner misuses the flag system. A frustrated learner flags every response as wrong even when AURA was correct. Or a learner never flags anything because they do not realise it is wrong. The flag system assumes rational, representative flagging behaviour.
MISSINGLearner accuracy of their own work. AURA's authenticity split tracks AI vs human contribution. But if a human writes something factually wrong, we log it as human content with full authenticity. The accuracy of the content itself is never measured.
FIXBehaviour-profile divergence detection. Compare declared profile settings against observed behaviour over 10+ sessions. If a learner declared "Light scaffolding preferred" but requests Tier 1 pre-write 90% of the time, flag the divergence and offer a profile update. This is automatic ground-truth checking on the learner's own data.
Pass 7Statistical accuracy: our sample sizes are wrongStatistics: sampling theory + Simpson's paradox
MISSINGThe flag system at low volume is noise. At 100 users, 3 flags on a response type means nothing statistically. At 10,000 users it means something. We have no minimum sample size threshold before accuracy patterns are treated as signal vs noise. Aaron could act on a false pattern.
MISSINGSimpson's paradox risk. AURA accuracy looks good overall but is terrible for one specific tier (say, Primary learners uploading handwritten documents). The aggregate accuracy metric hides the subgroup failure. We need accuracy broken down by: tier, document type, extraction method, rubric confidence band.
MISSINGSurvivorship bias in the accuracy data. Learners who give up and leave do not flag responses. The most damaging accuracy failures produce churn, not flags. We are measuring accuracy among retained users, which is a biased sample.
MISSINGNo A/B accuracy testing. We have one AURA prompt version. We have no way to test whether a different system prompt version produces more accurate responses. We cannot improve without comparison.
FIXBuild a minimum_signal_threshold into the accuracy dashboard. Do not surface accuracy patterns until n >= 30 instances. Break every accuracy metric by tier + document_type + extraction_method. Track session_ended_without_submission as a proxy accuracy signal for learners who churned.
Pass 8Privacy accuracy: the logging system itself creates riskPrivacy engineering + GDPR/Australian Privacy Act
MISSINGThe accuracy logs we are building contain learner data. ingestion_events stores extracted_fields which contains assessment content. aura_response_flags logs the response text hash. aura_context_snapshots stores profile data at call time. None of these tables were in the original three-layer data architecture. Where do they sit? Who can see them?
MISSINGThe correction_delta in ingestion_events stores what the learner changed. That delta reveals the gap between what was in the document and what the learner expected. That is sensitive academic data. It needs the same protection as Layer 1 (learner work content), not Layer 2 (telemetry).
MISSINGLog retention policy is undefined. How long do accuracy logs live? GDPR right to deletion applies. If a learner requests data deletion, do the accuracy logs get purged? If yes, we lose the training signal. If no, we are in violation.
MISSINGThe raw_text_hash is a SHA256 of document content. In theory a hash cannot be reversed. In practice, for short documents with predictable content (standard rubric formats), rainbow table attacks are possible. We are storing a weak pseudonymisation of document content.
FIXAssign every new log table a data_classification: operational (can be retained, aggregated, used for product improvement) or sensitive (must be deletable on request, never aggregated with identifying data). Add a privacy_review_required flag to the CLAUDE.md before any new table is created.
Pass 9Accuracy of the accuracy system: who watches the watchmenSecond-order systems thinking + meta-validation
MISSINGThe confidence_score calculation itself is unvalidated. We defined it as chars_extracted / estimated_chars_from_page_count. That formula has never been tested against ground truth. It could systematically over-report confidence for dense PDFs and under-report for sparse ones.
MISSINGThe write verification hash (JSON.stringify length + first 100 chars) is weak. Two different objects can have the same length and same first 100 characters. It is not a real integrity check. It is a comfort check.
MISSINGThe flag button requires cognitive load from the learner at the exact moment they are most confused and frustrated. The accuracy system depends on learner effort at the worst possible moment. Flagging rates will be low not because AURA is accurate but because flagging is hard.
MISSINGAaron has no dashboard. All of this logging builds up in Supabase but there is no interface to see it. An accuracy system that produces data nobody reads is not an accuracy system. It is a false sense of security.
FIXThree things: (1) Replace the confidence formula with a validated one: extract the same document twice with different pdfjs settings, compare outputs, use divergence as the confidence measure. (2) Replace the hash with a proper CRC32 of the full serialised object. (3) Build a minimal accuracy dashboard: one page, four numbers, updated daily: ingestion confidence p50/p95, flag rate per 100 calls, context truncation rate, write verification failure rate.
Pass 10Existential accuracy: what we are not even measuringFirst principles + unknown unknowns
MISSINGDid AURA's guidance actually help the learner learn? We measure whether the guidance was factually accurate to the document. We do not measure whether it produced better outcomes. A rubric-accurate Pareto Step delivered to a learner in crisis still fails. Accuracy of output is not accuracy of impact.
MISSINGAccuracy of AURA's pedagogical choices. AURA asks a Socratic question. The question is accurate to the rubric. But was it the right question for this learner at this moment? We have no outcome measure for pedagogical accuracy. Did the learner produce better writing after the Socratic exchange?
MISSINGAccuracy of the system prompt itself. The AURA v3 system prompt is our best work. But it has never been validated against outcomes. We do not know if the prohibited word list actually reduces shame spirals. We do not know if the three-nudge idle sequence produces better re-engagement than two nudges or four. The prompt is an untested hypothesis.
MISSINGAccuracy of who is actually using the account. A learner shares their login with a friend. The accuracy system logs everything against learner A's profile but learner B is the one interacting. Every accuracy measurement is now attributed to the wrong person.
FIXThe only real accuracy measure for an educational product is: did the learner perform better on their next assessment after using the system? Everything else is a proxy. Build the one metric that matters: link session_id to eventual submission_grade (where available) and track the correlation. That is the ground truth everything else serves.
